线程消息等待机制，以及睡眠

有时候多线程工作下，如果有其中的线程还没得到条件满足下可以考虑进入等待状态减少资源的消耗，待其他相关线程执行完后条件满足后再把进入等待中的线程唤醒进行继续运行，从而避免不必要的工作开销。

基础知识：notify( ) wait( ) notifyAll( ) 都是Object的方法，所以不管是对象锁还是类锁都能进行调用，但是因为唤醒与等待操作是对同一锁下的线程进行操作，所以必须在同一个锁下进行等待唤醒操作同步才会生效。

notify（）与wait（）的使用要注意以下几点:

  • 必须在同步块代码中调用，否则会抛出IllegalMonitorStateException异常
  • notify只能随机唤醒同一锁的一个线程
  • notifyAll可以唤醒同一锁的所有线程
  • notify调用后不会立即释放锁，会继续完成同步代码内的代码后再释放锁
  • wait调用后会立刻释放锁
  • wait(long)调用后时间到后会自动唤醒，当然也可以被主动notify唤醒
  • wait（）进入阻塞后，如果线程被中断会抛出异常InterruptedException
  • 注意代码段中 if，while 的使用选择
  • wait调用后，当线程被notify会继续接着之前wait后面的代码继续执行而不是重头执行同步快
  • 除非自己设定了特定的判断逻辑，否则notify很容易出死锁等问题，建议notifyAll唤醒比较安全

sleep（long）

  • Thread.sleep(0)会触发操作系统立刻重新进行一次CPU竞争
  • Thread.sleep（long）不会释放锁，wait则会
